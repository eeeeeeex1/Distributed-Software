# 秒杀系统设计文档

## 一、系统架构设计

### 1. 服务拆分与架构草图
采用微服务架构，按领域边界拆分为独立服务，通过注册中心（如 Nacos）实现服务治理，网关（Gateway）统一入口。

| 服务名称 | 核心职责 | 关键技术组件 |
|----------|----------|--------------|
| 用户服务 (User Service) | 注册、登录、身份认证、用户信息管理 | Spring Security / JWT |
| 商品服务 (Product Service) | 商品信息展示、商品详情、类目管理 | MySQL + Redis 缓存 |
| 库存服务 (Stock Service) | 库存扣减、库存回滚、库存查询 | 分布式锁 / 消息队列 |
| 订单服务 (Order Service) | 订单创建、订单状态流转、订单查询 | 事务管理 |
| 网关服务 (Gateway) | 路由转发、限流、统一鉴权 | Spring Cloud Gateway |

### 2. 核心 API 接口定义 (RESTful)
以用户和秒杀场景为例，接口需满足无状态、幂等性设计。

#### 用户服务
- POST /api/user/register：用户注册
- POST /api/user/login：用户登录（返回 JWT Token）
- GET /api/user/info：获取当前用户信息

#### 商品服务
- GET /api/product/list：商品列表分页查询
- GET /api/product/detail/{id}：商品详情

#### 秒杀 / 订单服务
- POST /api/seckill/{productId}：发起秒杀请求
- GET /api/order/list：查询用户订单列表

### 3. 数据库 ER 图设计
核心实体关系包含用户、商品、库存、订单四张核心表，需注意主键关联和外键约束。

#### 用户表 (t_user)
- 字段：id (PK), username, password, phone, avatar, create_time

#### 商品表 (t_product)
- 字段：id (PK), name, title, price, stock_count (冗余库存), img_url, status

#### 库存表 (t_stock)
- 字段：id (PK), product_id (FK), lock_stock (锁定库存), actual_stock (真实库存)

#### 订单表 (t_order)
- 字段：id (PK), order_no (唯一订单号), user_id, product_id, price, status (0 - 待支付，1 - 已支付，2 - 已取消), create_time

## 二、技术栈选型说明

### 1. 核心技术栈 (后端)
| 维度 | 选型 | 选型理由 |
|------|------|----------|
| 编程语言 | Java 11 | 生态成熟，高性能，适合高并发场景 |
| Web 框架 | Spring Boot 3.x | 快速开发，自动配置，简化开发 |
| 微服务框架 | Spring Cloud Alibaba | 组件丰富（Nacos、Sentinel、Seata） |
| 持久层 | MyBatis + MyBatis-Plus | 灵活编写 SQL，Plus 提供通用 CRUD |
| 数据库 | MySQL 8.0 | 开源成熟，支持事务，应对结构化数据 |
| 缓存 | Redis | 高性能读写，用于缓存商品信息、限流 |
| 服务治理 | Nacos | 注册中心 + 配置中心一体化 |
| 分布式事务 | Seata (AT 模式) | 解决微服务下的数据一致性问题 |
